# 文档写作风格指南

本文档规定指南文档的写作风格、术语规范和措辞要求。

## 语言规范

### 基本规则

- 文档正文使用简体中文
- 专业术语保留英文原文，首次出现时附中文说明
  - 正确：Schema（模式定义）是 Zod 的核心概念
  - 错误：模式定义是 Zod 的核心概念
  - 错误：Schema 是 Zod 的核心概念（缺少中文说明）
- 代码注释使用简体中文
- 代码中的变量名、函数名保持英文原文

### 中英文混排

- 中英文之间加空格：`使用 TypeScript 进行开发`
- 中文与代码标记之间加空格：`调用 \`parse()\` 方法`
- 标点符号使用中文标点（句号、逗号、冒号等）
- 括号内全英文时使用半角括号：`依赖注入 (Dependency Injection)`

## 术语规范

### 术语一致性

同一概念在全文中使用统一的术语，不混用同义词：

| 统一使用 | 避免使用 |
|---------|---------|
| 配置项 | 设置项、选项（作为名词时） |
| 函数 | 方法（仅 OOP 语境使用"方法"） |
| 参数 | 参量、入参 |
| 返回值 | 输出、返回结果 |
| 默认值 | 初始值、缺省值 |
| 属性 | 字段（仅在数据库语境使用"字段"） |
| 实例化 | 初始化（仅在配置语境使用"初始化"） |

### 版本相关术语

| 术语 | 含义 |
|------|------|
| 兼容 (compatible) | 与现有 API 共存，不影响已有用法 |
| 废弃 (deprecated) | 仍可用但建议迁移，将在未来版本移除 |
| 移除 (removed) | 已不可用，必须使用替代方案 |
| 破坏性变更 (breaking change) | 需要修改代码才能升级 |

## 措辞规范

### 操作步骤

使用祈使句，直接说明操作：

- 正确：`安装依赖包`
- 错误：`你需要安装依赖包`
- 错误：`让我们安装依赖包`

### 描述说明

使用陈述句，客观描述：

- 正确：`该函数返回一个 Promise 对象`
- 错误：`该函数会返回一个 Promise 对象`（避免"会"字冗余）
- 错误：`你应该使用 async/await`（避免说教语气）

### 提示和警告

使用明确的提示标记：

```markdown
> **提示：** 仅在 Node.js 14+ 环境下支持此特性。

> **警告：** 此 API 在 v3 中已废弃，请使用 `createRouter` 替代。
```

## 代码示例规范

### 命名风格

- 变量名使用有意义的英文单词：`userSchema` 而非 `s` 或 `schema1`
- 示例数据贴近实际：`"Alice"` 而非 `"foo"`，`30` 而非 `123`
- 避免使用纯演示性命名：`myVar`、`test1`、`example`

### 代码注释

```typescript
// 正确：注释说明意图
const result = schema.safeParse(input); // 不抛异常的验证方式

// 错误：注释复述代码
const result = schema.safeParse(input); // 调用 safeParse 方法
```

### 完整性

- 每个示例必须包含必要的 import 语句
- 示例代码可以独立运行，不依赖前文示例的变量
- 异步操作必须包含 `await` 或 `.then()` 处理

## 排版细节

### 代码块

- 始终标注语言类型：` ```bash `、` ```typescript `、` ```python `
- 终端命令使用 `$` 前缀或不使用前缀（全文统一）
- 输出结果用注释标注：

```bash
npm install zod
# added 1 package in 2s
```

### 链接

- 使用行内链接，不使用脚注式链接
  - 正确：`查看 [官方文档](https://...) 了解详情`
  - 错误：`查看官方文档^[1]^ 了解详情`
- 链接文本描述目标内容，不使用"点击这里"
- 不生成猜测的 URL，仅使用确认存在的链接

### 表格

- 表格必须有表头行
- 列对齐使用连字符分隔线
- 表格不适合展示复杂层级关系时，改用列表或代码块
