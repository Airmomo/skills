# 指南文档结构模板

本文档详细说明指南文档各章节的编写规范和内容要求。

## 章节编写规范

### 概述 (Overview)

**目标：** 让读者在 30 秒内判断这份文档是否适合自己。

**包含内容：**
- 一句话定义：这个工具/库是什么
- 核心价值：解决什么问题
- 适用场景：谁应该用、什么时候用
- 与同类工具的简要对比（可选）

**写作要求：**
- 控制在 200 字以内
- 不使用比喻，直接说明功能和用途
- 避免营销性语言

**模板：**

```markdown
## 概述 (Overview)

[工具名] 是一个 [类别] 工具/库，用于 [核心功能]。

主要特点：
- [特点 1]
- [特点 2]
- [特点 3]

适用于 [目标场景] 的开发场景。
```

### 快速开始 (Getting Started)

**目标：** 让读者在 10 分钟内跑通第一个示例。

**包含内容：**

1. **环境要求** — 列出前置依赖和版本要求
2. **安装** — 提供主流包管理器的安装命令
3. **最小可用示例** — 最简单的可运行代码
4. **验证安装** — 确认安装成功的检查方法

**写作要求：**
- 安装命令必须可复制执行
- 最小示例必须是完整可运行的代码
- 列出所有必要的前置条件

**模板：**

```markdown
## 快速开始 (Getting Started)

### 环境要求

- [依赖 1] >= [版本]
- [依赖 2] >= [版本]

### 安装

\`\`\`bash
# npm
npm install [package]

# yarn
yarn add [package]

# pnpm
pnpm add [package]
\`\`\`

### 最小可用示例

\`\`\`[lang]
// 完整可运行的代码
\`\`\`

### 验证安装

\`\`\`bash
[验证命令]
\`\`\`
```

### 核心概念 (Key Concepts)

**目标：** 帮助读者建立对工具的心智模型。

**编写方法：**
1. 从源码中提取核心抽象和关键类型
2. 每个概念用"是什么 → 为什么重要 → 怎么用"的三段式结构
3. 用简短代码片段说明每个概念的实际体现

**写作要求：**
- 概念数量控制在 3-7 个
- 每个概念控制在 200 字以内
- 概念之间按依赖关系排序（先讲被依赖的）

**概念描述模板：**

```markdown
### [概念名]

[概念名] 是 [定义]。用于解决 [问题]。

关键特征：
- [特征 1]
- [特征 2]

示例：

\`\`\`[lang]
// 体现该概念的代码片段
\`\`\`
```

### 配置参考 (Configuration Reference)

**目标：** 作为查阅手册，读者可以快速找到需要的配置项。

**写作要求：**
- 以表格形式呈现
- 必须包含：配置项名称、类型、默认值、说明
- 按功能分组，不按字母排序
- 必须与源码中的类型定义/默认值交叉验证

**表格模板：**

```markdown
## 配置参考 (Configuration Reference)

### [功能组 1]

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `option1` | `string` | `"default"` | 功能说明 |
| `option2` | `number` | `8080` | 功能说明 |
| `option3` | `boolean` | `false` | 功能说明 |
```

### 示例 (Examples)

**目标：** 覆盖 80% 的常见使用场景。

**结构：**
1. **基础用法** — 最常见的使用方式
2. **进阶用法** — 组合多个功能的使用方式
3. **常见场景** — 按实际业务场景组织的示例

**写作要求：**
- 每个示例必须是完整可运行的代码
- 每个示例前说明该示例解决什么问题
- 代码中包含关键注释
- 示例之间由浅入深排列

**示例模板：**

```markdown
### [示例标题]

**场景：** [这个示例解决什么问题]

\`\`\`[lang]
// 完整可运行代码
// 关键步骤注释
\`\`\`
```

### 常见问题 (FAQ)

**目标：** 减少读者在遇到问题时的排查时间。

**写作要求：**
- 问题从实际使用中收集，不编造
- 每个回答包含原因分析和解决方法
- 按问题发生频率排序

**FAQ 模板：**

```markdown
## 常见问题 (FAQ)

### [问题描述]

**原因：** [导致问题的根本原因]

**解决方法：**

\`\`\`[lang]
// 解决代码
\`\`\`
```

### 参考资源 (References)

**目标：** 提供进一步学习的途径。

**写作要求：**
- 列出官方文档、GitHub 仓库等权威链接
- 如有推荐的学习路径，简要标注适合的读者层级
- 链接必须可访问，不使用猜测的 URL

```markdown
## 参考资源 (References)

- [官方文档](链接) — 完整 API 参考
- [GitHub 仓库](链接) — 源码和 Issue 追踪
- [入门教程](链接) — 适合初学者的系统教程
```

## 交叉验证清单

生成配置参考章节时，使用以下清单确保准确性：

- [ ] 函数/方法签名与源码一致（参数名、类型、返回值）
- [ ] 配置项默认值与源码中的定义一致
- [ ] 导入路径与源码文件结构一致
- [ ] 代码示例中引用的类名/方法名在源码中存在
- [ ] 版本号与当前最新版本一致

## 文档排版规范

1. **标题层级：** `#` 文档标题 → `##` 一级章节 → `###` 二级章节 → `####` 三级章节
2. **英文副标题：** 所有一级章节（`##`）统一添加英文副标题，格式为 `## 中文标题 (English Title)`
3. **代码块：** 始终标注语言类型（`bash`、`typescript`、`python` 等）
4. **表格：** 仅用于结构化数据（配置项、API 参数等），不用于流程说明
5. **列表：** 操作步骤用有序列表，特征/要点用无序列表
6. **强调：** 用反引号标记代码/命令，用加粗标记关键术语

## 文档更新与维护

### 版本标注规范

每份文档开头须注明适用的库版本范围：

```markdown
> 本指南基于 [库名] v[版本号] 编写，最后更新：[日期]
```

### 文档更新触发条件

以下情况需要更新文档：

| 触发条件 | 更新范围 |
|---------|---------|
| 库发布 major 版本 | 全面审查，更新所有章节 |
| 库发布 minor 版本 | 检查新功能，更新核心概念和示例 |
| API 签名变更 | 更新配置参考和相关示例 |
| 默认值变更 | 更新配置参考表格 |
| 废弃 (deprecated) API | 在相关章节标注废弃说明和替代方案 |

### 更新流程

1. 确认当前库版本与文档版本
2. 对照 changelog 识别变更点
3. 按触发条件表确定更新范围
4. 更新文档内容
5. 重新执行交叉验证清单
6. 更新文档开头的版本号和日期

### 非公开项目/内部工具处理

当目标库没有公开 GitHub 仓库时（内部 SDK、私有包等）：

1. **信息收集调整：**
   - 跳过 zread MCP（无法访问私有仓库）
   - 优先使用项目内部的 README、JSDoc/TSDoc 注释、类型定义文件
   - 从本地代码库直接读取源码作为验证依据

2. **交叉验证调整：**
   - 以本地源码为验证基准（替代远程仓库源码）
   - 通过类型定义文件（`.d.ts`、`types/` 目录）确认 API 签名
   - 通过测试文件确认 API 的实际用法

3. **文档标注：**
   - 在文档开头注明"本文档基于 [内部库名] 内部源码编写"
   - 不添加公开链接，改用内部文档系统链接
   - 如适用，注明文档的内部可见范围
