# Guide Builder

通过 MCP 文档工具查阅最新官方文档，结合实际源码交叉验证，构建结构完整、内容准确的技术指南文档。

## 与 dev-doc-generator 的区分

- **本技能（quick-guide-builder）**：为某个库、框架、工具撰写**技术操作指南**（Getting Started、使用教程、配置参考等）
- **dev-doc-generator**：基于已有参考项目为新项目生成**开发蓝图文档**（架构设计、技术方案等）

## 何时使用

当用户需要为某个库、框架、工具或技术主题创建结构化的指南文档时使用本技能。适用于从零开始撰写技术文档，也适用于更新或改进已有文档。

## 工作流程

### 第一步：明确文档范围

与用户确认文档主题和范围：

1. **目标主题** — 要撰写哪个库/框架/工具的指南
2. **目标读者** — 面向初学者、中级开发者还是高级用户
3. **重点章节** — 是否有特别需要详述的章节
4. **文档深度** — 简要参考手册还是详尽教程
5. **输出位置** — 文档保存路径

### 第二步：收集文档资料

使用可用的 MCP 文档工具查阅最新官方文档，按以下优先级获取信息：

**优先级 1：Context7 MCP** — 查询库的最新文档和代码示例
- 先调用 `resolve-library-id` 解析库 ID
- 再调用 `query-docs` 获取具体主题的文档内容
- 每个主题最多调用 3 次，避免重复查询

**优先级 2：zread MCP** — 直接读取 GitHub 仓库的源码和文档
- 使用 `get_repo_structure` 了解仓库结构
- 使用 `read_file` 读取关键源码文件
- 使用 `search_doc` 搜索特定主题

**优先级 3：Web Search** — 补充 Context7 和 zread 未覆盖的内容

**降级策略：**

| 情况 | 处理方式 |
|------|---------|
| Context7 找不到目标库 | 跳过，转用 zread 或 Web Search |
| zread 仓库访问受限（私有仓库） | 转用 Web Search，并在文档中标注"未能验证源码" |
| 仅 Web Search 可用 | 正常收集，但第三步交叉验证需标注"基于文档推断，未经源码验证" |
| 三个信息源均无法获取 | 通知用户无法收集到足够资料，询问是否继续基于已有知识生成（标注局限性） |

### 第三步：交叉验证

将收集到的文档内容与实际源码进行交叉验证：

**验证内容：**
1. **API 签名验证** — 确认函数/方法的参数、返回值与源码一致
2. **配置项验证** — 确认配置字段、默认值与源码定义一致
3. **示例验证** — 确认代码示例中的导入路径、类名、方法名确实存在
4. **版本一致性** — 确认文档引用的 API 在当前版本中可用

**验证方法：**
- 在源码中搜索函数定义和类型声明
- 读取源码中的关键实现文件
- 确认文件路径是否存在

**交叉验证清单（每项必须逐条检查）：**

- [ ] 函数/方法签名与源码一致（参数名、类型、返回值）
- [ ] 配置项默认值与源码中的定义一致
- [ ] 导入路径与源码文件结构一致
- [ ] 代码示例中引用的类名/方法名在源码中存在
- [ ] 版本号与当前最新版本一致

**验证失败处理：**

| 失败类型 | 处理方式 |
|---------|---------|
| API 签名不匹配 | 以源码为准，更新文档并在文档中标注实际版本 |
| 配置项缺失 | 标注为"该配置项仅在特定版本/环境下可用" |
| 示例代码无法验证 | 标注为"未经源码验证"并保留，同时尝试从官方文档获取替代示例 |
| 无法访问源码 | 在文档开头注明"本文档基于 [信息源] 编写，部分内容未经源码级验证" |

### 第四步：生成指南文档

按照以下标准结构生成 Markdown 指南文档：

```markdown
# [主题名称] 指南

## 概述 (Overview)

简要介绍：是什么、解决什么问题、适用场景。控制在 200 字以内。

## 快速开始 (Getting Started)

### 环境要求
### 安装
### 最小可用示例
### 验证安装

## 核心概念 (Key Concepts)

### 概念 1
### 概念 2
### 概念 N

## 配置参考 (Configuration Reference)

以表格形式列出所有配置项：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|

## 示例 (Examples)

### 基础用法
### 进阶用法
### 常见场景

## 常见问题 (FAQ)

收集实际使用中的常见问题和解决方案。

## 参考资源

- 官方文档链接
- GitHub 仓库链接
```

**篇幅控制建议：**

| 章节 | 建议篇幅 | 说明 |
|------|---------|------|
| 概述 | 100-200 字 | 简明扼要，不展开技术细节 |
| 快速开始 | 30-80 行代码 | 最小可用示例 + 安装命令 |
| 核心概念 | 3-7 个概念，每个 100-200 字 | 按依赖关系排序 |
| 配置参考 | 按需 | 按功能分组，表格呈现 |
| 示例 | 3-6 个 | 基础→进阶→场景，由浅入深 |
| 常见问题 | 3-10 条 | 按发生频率排序 |

### 第五步：审查与输出

生成文档后执行自检，逐项通过以下质量门禁：

**质量门禁（全部通过方可输出）：**

- [ ] **准确性** — 所有代码示例经过源码验证或标注验证状态
- [ ] **完整性** — 核心章节（概述、快速开始、核心概念、示例）齐全
- [ ] **一致性** — 术语、命名风格全文统一；英文副标题格式统一
- [ ] **可读性** — 排版层级清晰，代码块标注语言类型
- [ ] **交叉验证** — 第三步清单中的所有项已检查
- [ ] **验证标注** — 未能验证的内容已标注局限性说明

**门禁未通过时的处理：**
- 准确性未通过：回退到第三步重新验证，或标注为"待验证"
- 完整性未通过：补充缺失章节，确实无法补充的标注"暂缺"
- 一致性/可读性未通过：直接修正后重新检查

将最终文档保存到用户指定的路径。

## 文档质量标准

- 每个代码示例必须经过源码验证，无法验证的须标注"待验证"
- 配置参考必须与源码中的类型定义一致
- 使用祈使句描述操作步骤
- 中文文档中，专业术语保留英文原文并附中文说明
- 每个章节开头用 1-2 句话概括本章内容
- 代码块始终标注语言类型（`bash`、`typescript`、`python` 等）

## 示例输出片段

以下展示最终文档各章节的预期风格：

**概述示例：**

```markdown
## 概述 (Overview)

Zod 是一个 TypeScript 优先的 schema 声明和验证库，用于在运行时验证数据结构。

主要特点：
- TypeScript 类型推断 — 无需手动定义接口
- 链式 API — 灵活组合验证规则
- 零依赖 — 体积小，适合任何项目

适用于需要验证外部输入（API 响应、表单数据、配置文件）的 TypeScript 项目。
```

**快速开始示例：**

```markdown
## 快速开始 (Getting Started)

### 环境要求

- Node.js >= 14
- TypeScript >= 4.5（推荐）

### 安装

\`\`\`bash
npm install zod
\`\`\`

### 最小可用示例

\`\`\`typescript
import { z } from "zod";

const UserSchema = z.object({
  name: z.string(),
  age: z.number().min(0),
});

const user = UserSchema.parse({ name: "Alice", age: 30 });
// { name: "Alice", age: 30 }
\`\`\`
```
