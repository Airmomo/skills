# 执行路线设计模式

## 核心思想

执行路线是一棵**增量构建树**：主干是开发顺序，每个分支是一个功能节点，每个叶节点完成后项目处于可运行状态。项目从一棵幼苗逐渐长成完整的大树。

## 依赖排序策略

所有模块按依赖关系分层，从底层到上层依次开发：

```
┌─────────────────────────────────────────────┐
│ L5 · 体验优化（动画、无障碍、国际化）        │
├─────────────────────────────────────────────┤
│ L4 · 辅助功能（搜索、筛选、通知、导出）      │
├─────────────────────────────────────────────┤
│ L3 · 核心业务流程（CRUD、工作流、状态机）     │
├─────────────────────────────────────────────┤
│ L2 · 基础设施（路由、状态管理、网络层、存储） │
├─────────────────────────────────────────────┤
│ L1 · 项目骨架（框架初始化、目录结构、构建配置）│
└─────────────────────────────────────────────┘
```

**排序原则：**
- 被依赖的模块优先开发
- 同层模块按核心流程的执行顺序排列
- 外围功能（可插拔的模块）放在最后

## 节点拆分粒度

### 判断标准

一个节点应该满足以下条件：

| 条件 | 说明 |
|------|------|
| 单一可验证 | 完成后可以独立测试验证 |
| 有明确的输入输出 | 能清晰地描述"做了什么" |
| 时间可控 | 单个节点的开发量不过大（建议 1-3 天） |
| 可独立运行 | 完成后项目可编译启动，不依赖后续节点 |
| 有验收标准 | 能用一句话描述"怎样算完成" |

### 拆分示例

以"会话管理"模块为例：

| 粒度过粗 | 合适粒度 | 粒度过细 |
|----------|----------|----------|
| 会话管理 | N01 · 会话列表展示 | 会话列表 HTML 结构 |
| | N02 · 新建会话 | 会话列表 CSS 样式 |
| | N03 · 会话切换与恢复 | 会话列表点击事件绑定 |
| | N04 · 会话删除与归档 | 新建会话弹窗组件 |

### 拆分时机

- **何时拆细**：当节点涉及多个独立交互或多个 UI 页面时
- **何时合并**：当两个功能紧密耦合、分开后无法独立测试时
- **何时保持**：当功能天然是一个原子操作时（如"项目初始化"）

## 验收标准编写指南

验收标准是判断节点完成的唯一依据，需要**具体、可操作、可验证**。

### 格式模板

```
验收方式：
1. 启动项目，执行 <操作>
2. 观察 <现象>
3. 确认 <结果>
```

### 编写原则

| 原则 | 正确示例 | 错误示例 |
|------|----------|----------|
| 描述具体操作 | "点击新建按钮，输入标题，点击确认" | "能创建会话" |
| 描述可见结果 | "列表中出现新会话项，标题为输入值" | "功能正常" |
| 包含边界情况 | "输入空标题时显示错误提示" | "有基本校验" |
| 不依赖后续节点 | "会话数据保存到本地存储" | "会话与服务器同步" |

### 验收标准类型

| 类型 | 适用场景 | 示例 |
|------|----------|------|
| 视觉验收 | UI 组件 | "页面显示登录表单，包含用户名和密码输入框" |
| 功能验收 | 业务逻辑 | "提交表单后跳转到主页，localStorage 中存在 token" |
| 数据验收 | 数据处理 | "导入 CSV 文件后，数据库中新增 N 条记录" |
| 集成验收 | API 对接 | "调用接口返回 200，响应体包含预期字段" |
| 性能验收 | 性能优化 | "列表渲染 1000 条数据，滚动帧率 ≥ 30fps" |

## 常见项目类型的路线模板

### Web 前端项目

```
N01 · 项目骨架（框架 + 目录 + 构建）
N02 · 路由与布局（页面框架 + 导航）
N03 · 状态管理与网络层（API 封装 + 全局状态）
N04 · 核心页面 · 列表展示（数据获取 + 渲染）
N05 · 核心页面 · 创建与编辑（表单 + 提交）
N06 · 核心页面 · 详情与删除（查看 + 确认删除）
N07 · 用户认证（登录 + 权限守卫）
N08 · 辅助功能（搜索 + 筛选 + 分页）
N09 · 体验优化（加载状态 + 错误处理 + 动画）
```

### 后端服务项目

```
N01 · 项目骨架（框架 + 目录 + 配置管理）
N02 · 数据库与模型（表结构 + ORM + 迁移）
N03 · 核心 API · CRUD（增删改查接口）
N04 · 认证与授权（JWT + 中间件 + 角色权限）
N05 · 业务逻辑层（服务层 + 事务处理）
N06 · 集成与通信（第三方 API + 消息队列）
N07 · 辅助功能（日志 + 监控 + 健康检查）
N08 · 部署配置（Docker + CI/CD + 环境变量）
```

### 全栈项目

```
N01 · 项目骨架（Monorepo + 前后端目录 + 共享类型）
N02 · 后端骨架（数据库 + 配置 + 基础 API 框架）
N03 · 前端骨架（框架 + 路由 + 布局 + 状态管理）
N04 · 数据模型（共享类型定义 + 数据库迁移）
N05 · 核心 API（后端 CRUD + 前端对接）
N06 · 核心页面（前端列表 + 表单 + 详情）
N07 · 用户认证（后端 JWT + 前端登录 + 权限流）
N08 · 实时通信（WebSocket / SSE + 前端订阅）
N09 · 辅助功能（搜索 + 通知 + 导出）
N10 · 部署与运维（Docker Compose + CI/CD）
```

### CLI 工具项目

```
N01 · 项目骨架（框架 + CLI 入口 + 参数解析）
N02 · 核心命令 · 初始化（交互式创建项目）
N03 · 核心命令 · 执行（主要业务逻辑）
N04 · 配置管理（配置文件读写 + 默认值）
N05 · 输出与日志（格式化输出 + 日志级别）
N06 · 辅助命令（版本、帮助、检查）
N07 · 分发（打包 + npm 发布 / 二进制构建）
```

## 路线与文档的映射关系

执行路线中的每个节点都应该能映射到具体的文档章节：

```
执行路线（README.md）
├── N01 · 项目骨架 ────────→ 技术架构文档 · 项目初始化章节
├── N02 · 路由与布局 ──────→ UI 与布局设计 · 布局结构章节
├── N03 · 状态管理层 ──────→ 技术架构文档 · 状态管理章节
├── N04 · 列表展示 ────────→ 功能模块设计 · 列表模块章节
├── N05 · 创建与编辑 ──────→ 功能模块设计 · 表单模块章节
└── ...
```

**映射原则：**
- 一个节点可以映射到多个文档的相关章节
- 一个文档可以服务多个节点
- 节点的详细设计直接链接到对应文档章节
- 验收标准不应与文档内容重复，而是引用文档中的具体规格
