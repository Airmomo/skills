# 文档结构模板与组合规则

## 核心原则

文档结构**不是固定的**，根据用户选择的模块动态组合。以下是各模块对应的文档模板。

## 必有文档（所有场景）

### README.md — 主文档

```markdown
---
name: 项目文档总览
description: <项目名称>的主文档，提供项目概述与所有子文档索引
roles: [ProductManager, Architect, ProjectManager, Engineer, QA]
---

# <项目名称> — 项目开发文档

## 项目背景
<项目目的与核心价值>

## 团队角色
| 角色 | 职责 |
|------|------|
| ProductManager | ... |

|------|------|------|-------------|
| 01 | [链接] | 描述 | 角色 |

## 参考实现代码
<reference/ 目录说明>

## 阅读顺序建议
```

### 产品需求文档

包含选中模块的功能需求、非功能需求、MVP 范围。
- 明确"不在范围内"的功能
- 非功能需求需要可量化的指标
- 用户流程用有序列表

### 技术架构文档

包含选中模块的技术选型、架构设计、依赖清单。
- 依赖清单按分类组织，每项标注包名、版本、用途、安装命令
- 进程架构用 ASCII 图表
- 通信边界用表格明确

### 开发规范文档

编码规范、测试策略、Git 工作流、构建发布。
- 与技术栈匹配的工具链推荐

## 按需文档（根据模块选择）

### UI 与布局设计（选中 UI 相关模块时）

```markdown
---
name: UI 与布局设计
description: 定义客户端的整体布局结构、页面路由与主题系统
roles: [ProductManager, Engineer]
---
```

包含：布局结构、页面路由、组件层级、主题系统（CSS 变量、明暗模式）、响应式适配。

### 协议集成文档（选中协议/通信模块时）

```markdown
---
name: <协议名称>集成
description: 定义协议规范、连接生命周期、消息流转与权限处理
roles: [Architect, Engineer]
---
```

包含：协议概述、消息类型、完整发送/接收流程、流式处理、权限机制、超时与重连。

### 数据模型与存储（选中数据/存储模块时）

```markdown
---
name: 数据模型与存储
description: 定义数据库表结构、配置存储与状态管理策略
roles: [Architect, Engineer]
---
```

包含：数据库表结构、Repository 模式、配置存储方案、前端状态管理、数据迁移。

### 功能模块设计（选中功能模块时）

```markdown
---
name: 功能模块设计
description: 定义各功能模块的职责边界与接口设计
roles: [Architect, ProjectManager]
---
```

包含：每个选中模块的职责、接口、交互设计。模块较少时可合并为一个文档；模块较多时可按功能域分拆为多个子文档。

### 已知问题与解决方案（选中"已知问题规避"或用户明确要求时）

```markdown
---
name: 已知问题与解决方案
description: 汇总已发现的问题，为新项目提供规避指南
roles: [QA, Engineer]
---
```

每个问题包含：严重度、位置、现象、根因、解决方案。

## 文档合并与分拆规则

| 模块数 | 建议文档数 | 策略 |
|--------|-----------|------|
| 1-2 个 | 3-4 个 | 必有文档 + 按需文档合并 |
| 3-5 个 | 5-7 个 | 必有文档 + 每类按需文档独立 |
| 6+ 个 | 7-10 个 | 必有文档 + 按需文档独立 + 功能模块可分拆 |

**合并原则**：内容关联紧密的模块可合并为一个文档（如 UI 布局 + 主题系统）。

**分拆原则**：单个文档超过 200 行时考虑分拆（如功能模块按域分拆为多个子文档）。

## 参考代码目录规则

```
reference/
├── README.md          # 索引文件（必须）
├── <module-a>/        # 按选中模块组织
└── <module-b>/
```

- 仅提取用户选中模块的代码
- README.md 说明每个文件的用途、行数和重点关注内容
- 文档中引用 `reference/` 路径，不引用原始项目源码路径
