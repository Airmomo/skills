# Dev Doc Generator

基于已有参考项目或用户需求，为新项目生成结构清晰、独立可用的开发文档。核心原则：**先确认需求，再决定文档结构**。

## 何时使用

当用户需要生成开发文档时触发。典型场景：

- 用户说"帮我基于 XX 项目写开发文档"
- 用户希望将已有项目的技术方案提炼为独立文档
- 用户只需要某个项目的部分模块的文档
- 用户从零描述需求，需要生成完整开发方案

## 如何使用

### 第一步：需求探索与模块确认

这是最关键的步骤。不要假设用户需要什么，通过交互确认需求。

**1.1 判断需求来源**

| 来源 | 处理方式 |
|------|----------|
| 用户指定了参考项目 | 先快速扫描项目，提取可实现的模块清单 |
| 用户只描述了需求 | 根据描述推断所需模块 |
| 两者都有 | 结合参考项目与用户描述综合判断 |

**1.2 扫描参考项目（如已指定）**

启动 1 个快速探索 Agent，对参考项目做高层扫描：

- 读取项目 README、package.json、docs/ 目录
- 识别项目的功能模块边界（如会话管理、消息渲染、协议通信、数据存储等）
- 列出每个模块的一句话描述和对应的源码路径
- 识别模块间的依赖关系

**1.3 生成模块选择清单**

基于扫描结果，向用户呈现模块选择清单。使用 AskUserQuestion 工具，将功能模块作为选项列出，支持多选。

清单格式示例：

| 选项 | 模块名称 | 描述 |
|------|----------|------|
| ☐ | 会话管理 | 对话的创建、列表、搜索、恢复、删除 |
| ☐ | 消息渲染 | Markdown、工具调用、思考过程、流式展示 |
| ☐ | 协议通信 | ACP/JSON-RPC、连接生命周期、权限处理 |
| ☐ | 数据存储 | 数据库设计、配置管理、状态管理 |
| ☐ | UI 框架 | 布局、路由、主题、响应式 |
| ☐ | ... | ... |

同时提示用户："以上是基于参考项目提取的模块，也可以补充其他模块需求。"

**1.4 补充确认**

用户选择模块后，针对所选模块追问关键细节：

- 目标平台/技术栈是否有硬性要求？
- 是否需要裁剪功能（如"仅支持 Claude CLI，不需要其他 Agent"）？
- 文档交付给什么团队？角色划分是什么？
- 文档是否需要完全独立于参考项目（提取参考代码到文档目录）？

**需求明确后，记下以下决策：**

- 选中模块列表
- 每个模块的功能裁剪范围
- 团队角色定义
- 文档独立性要求
- 技术栈约束（如有）

### 第二步：深度探索选中模块

基于第一步确认的模块范围进行深度分析。根据是否有参考项目，采取不同策略：

**有参考项目时**：启动探索 Agent 深度分析源码。

探索策略按模块数量动态分配：

| 选中模块数 | 探索策略 |
|------------|----------|
| 1-3 个 | 启动 1 个 Agent，集中探索选中模块 |
| 4-6 个 | 启动 2 个 Agent，按职责分组（如 UI 组 + 后端组） |
| 7+ 个 | 启动 3 个 Agent，三路并行 |

每个 Agent 的探索指令中**仅包含用户选中的模块**，跳过无关代码，减少 token 消耗。

探索重点关注：
- 选中模块的完整实现细节
- 模块间的依赖关系（仅探索已选中模块之间的依赖）
- 模块相关的已知问题
- 依赖包清单

**无参考项目时**：基于用户需求进行模块分析和架构推演。

- 列出每个选中模块的核心职责和对外接口
- 推导模块间的依赖关系和数据流向
- 根据目标技术栈推断合理的依赖包清单
- 识别潜在的架构风险和技术难点
- 从参考项目中提炼已知问题的模式（如用户提及类似项目可类比）

### 第三步：动态设计文档结构

**文档结构不是固定的**，根据选中的模块动态组合：

**必有文档**（所有场景都包含）：
- `README.md` — 主文档索引
- `产品需求文档` — 选中模块的功能需求
- `技术架构文档` — 选中模块的技术方案（含依赖清单）
- `开发规范文档` — 编码规范、测试策略、Git 工作流

**按需生成文档**（仅当用户选了对应模块时生成）：

| 选中模块 | 生成的文档 |
|----------|-----------|
| UI 框架 / 布局 / 主题 | `UI 与布局设计` |
| 协议通信 / Agent 集成 | `协议集成文档` |
| 数据存储 / 数据库 | `数据模型与存储` |
| 消息渲染 / 虚拟列表 | `功能模块设计`（合并到功能模块文档） |
| 已知问题规避 | `已知问题与解决方案` |

**文档分拆原则：**
- 每个子文档围绕单一关注点，不交叉引用其他子文档
- 子文档间通过主文档索引关联
- 模块较少时可合并（如 3 个模块可能只需 3-4 个文档）
- 模块较多时可进一步分拆（如协议层按收发方向分拆）

### 第四步：设计执行路线

基于第二步的深度探索结果和第三步的文档结构，设计一条从零到完整项目的**增量开发路线**。

**这是整个文档体系的主轴**——所有模块文档围绕路线节点展开，像一棵树一样生长。

**4.1 排序模块依赖**

将用户选中的模块按依赖关系分层排序：

```
L1 · 项目骨架    → 框架初始化、目录结构、构建配置
L2 · 基础设施    → 路由、状态管理、网络层、存储
L3 · 核心业务    → CRUD、工作流、状态机
L4 · 辅助功能    → 搜索、筛选、通知、导出
L5 · 体验优化    → 动画、无障碍、国际化
```

- 被依赖的模块优先开发
- 同层模块按核心流程的执行顺序排列

**4.2 拆分开发节点**

将每个模块拆分到具体功能粒度（如"会话列表展示"而非仅"会话管理"）。每个节点需满足：

- 完成后项目可编译、可启动、可测试
- 有明确的输入输出和验收标准
- 开发量控制在合理范围内（建议 1-3 天）

**4.3 生成路线表**

路线写入 README.md 的「执行路线」章节，格式：

| 序号 | 节点 | 前置依赖 | 验收标准 |
|------|------|----------|----------|
| N01 | 项目骨架搭建 | 无 | 项目可启动，显示空白页面 |
| N02 | ... | N01 | ... |

每个节点在「节点详情」中展开：目标、涉及模块、详细设计链接、验收方式（具体操作步骤）。

**4.4 路线与文档映射**

确保路线中每个节点都能映射到具体的文档章节，文档索引中标注每个文档服务的路线节点。

> 路线设计模式的详细参考见 `references/roadmap-patterns.md`。

### 第五步：编写文档

每个文档包含统一的 YAML 头部字段：

```
---
name: 文档名称
description: 一句话描述文档内容和作用
roles: [负责角色1, 负责角色2]
---
```

README.md 中的「执行路线」章节是文档体系的**主干**，按路线顺序组织内容：

```
README.md（执行路线）
├── N01 · 骨架 ──────→ 技术架构文档 · 初始化章节
├── N02 · 路由 ──────→ UI 设计文档 · 布局章节
├── N03 · 数据层 ────→ 数据模型文档 · 表结构章节
└── ...
```

**编写规范**

- 使用中文编写（除非用户另有要求）
- 保持内容简洁，不引入过多注释和代码块
- 表格优先用于对比性信息（依赖清单、配置项、问题列表等）
- ASCII 图表用于展示架构、流程和布局
- 关键决策需说明"为什么"而不只是"是什么"
- **仅编写用户选中模块的内容**，不包含未选中的功能
- 节点详情中的验收方式要具体可操作（参见 `references/roadmap-patterns.md` 中的验收标准指南）

### 第六步：提取参考实现代码（有参考项目时）

> 此步骤仅在用户指定了参考项目时执行。无参考项目时跳过。

将文档中需要引用的实现代码提取到独立的 `reference/` 目录：

```
reference/
├── README.md          # 参考代码索引与说明
├── <module-a>/        # 按选中的模块组织
│   ├── core.ts
│   └── utils.ts
└── <module-b>/
    └── adapter.ts
```

- 文档中引用 `reference/` 路径，不引用原始项目的源码路径
- `reference/README.md` 说明每个文件的用途、行数和重点关注内容
- **仅提取用户选中模块相关的参考代码**，不提取无关模块
- 确保文档完全独立于参考项目，可脱离原始代码库阅读

### 第七步：补充依赖清单

涉及技术选型的文档必须包含完整的依赖清单表：

| 列 | 说明 |
|----|------|
| 包名 | 包全名 |
| 版本 | 主版本号 |
| 用途 | 一句话说明 |
| 安装 | 安装命令（如 `bun add xxx`） |

仅列出用户选中模块涉及的依赖，不列出无关依赖。

### 第八步：验证与输出

检查项：
- 文档内容严格限于用户选中的模块范围
- 所有文档的 YAML 头部字段完整
- 子文档间无直接引用，仅通过主文档索引关联
- 无对原始项目源码路径的直接引用（统一指向 `reference/`）
- 依赖清单包含包名和安装命令
- 已知问题（如有）均附带解决方案
- **执行路线完整**：每个节点有明确的验收标准和前置依赖
- **路线可增量执行**：每个节点完成后项目处于可运行状态
- **路线与文档映射正确**：每个节点都能链接到对应的文档章节

## 关键原则

**先问后做**：不假设用户需要完整项目文档，先确认模块范围再动手。

**路线先行**：执行路线是文档体系的主轴。所有模块文档围绕路线节点展开，像一棵树——主干是路线，枝叶是各模块的详细方案。每个叶节点（功能点）都是完善可运行的，直到最后长成一棵完整的树、一个完整可运行的项目。

**按需裁剪**：文档结构、参考代码、依赖清单都只覆盖用户选中的模块。

**增量可验**：路线中每个节点完成后，项目必须处于可编译、可运行、可测试的状态。不依赖后续节点即可验证当前节点的完成度。

**保持独立**：最终文档体系应可脱离参考项目独立阅读和使用。
