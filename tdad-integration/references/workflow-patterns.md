# TDAD 工作流模式详解

> 核心工作流（完整循环流程、子代理启动、失败处理）参见 SKILL.md。本文档提供 SKILL.md 未覆盖的进阶模式。

## 子代理隔离的原理

经典 TDD 在单次对话中执行时存在**上下文污染**——编写测试时的分析思路会渗透到实现阶段，AI 不自觉地按测试思路编码而非独立从需求出发。TDAD 通过子代理隔离解决此问题：每个阶段在独立上下文窗口中运行，测试编写代理看不到实现代码，实现代理只看到测试错误输出。

详见 SKILL.md「核心原则 — 子代理隔离」。

## 进阶协调模式

### 回退修复模式

当 GREEN 阶段无法一次性通过所有测试时：

```
[GREEN 代理] → 部分失败 → [回传失败输出] → [GREEN 代理重试]
     ↑                                              │
     └────────────── 最多 3 次 ─────────────────────┘
```

### 混合模式

根据任务复杂度灵活组合：
- 简单任务：RED → GREEN（跳过 REFACTOR）
- 重构任务：仅 REFACTOR
- Bug 修复：分析失败 → GREEN（补充测试后）

### 并行测试生成

对于无依赖关系的多个独立功能，可并行启动多个 RED 代理，然后依次进入 GREEN 和 REFACTOR。注意：并行模式要求功能间无共享状态。

## 渐进式采用路线图

### 阶段 1：测试先行意识

不改变现有工具链，只在每次开发前先让 AI 编写测试。

- 保持交互反馈模式
- 让测试成为反馈的一部分
- 零额外配置，立即可行

### 阶段 2：技能 + 代理配置

创建 `.claude/agents/` 下的 TDAD 代理文件和 `.claude/skills/` 下的协调技能。

- 1-2 周配置
- 从简单的纯函数开始验证
- 逐步扩展到复杂场景

### 阶段 3：全自主闭环

Headless 模式运行完整 TDAD 循环。

- 开发者工作流变为：描述需求 → 等待 AI 完成 → 审查最终结果
- CI 集成，自动回归检测
- 多代理协作处理复杂任务

## Hooks 自动化（可选进阶）

通过 Claude Code 的 Hooks 机制自动激活 TDAD 技能：

```json
{
  "hooks": {
    "UserPromptSubmit": [{
      "matcher": "add|implement|create|build|feature",
      "command": "evaluate-tdad-skill"
    }]
  }
}
```

此配置将技能激活率从 ~20% 提升到 ~84%。

## 与 Vibe Coding 的关系

TDAD 不是对 Vibe Coding 的否定，而是其进化形态：

| 维度 | Vibe Coding | TDAD |
|------|-------------|------|
| 需求描述 | 自然语言 | 自然语言 + 测试契约 |
| 正确性保障 | 无 | 测试即规格 |
| AI 自主度 | 高（但无约束） | 高（在测试约束内） |
| 适用项目规模 | 中小型 | 不限 |
| 撞墙时间 | ~3 个月 | 不撞墙（测试保护） |

TDAD 保留了「自然语言描述意图」的核心理念，增加了「测试作为可执行契约」的质量保障层。
